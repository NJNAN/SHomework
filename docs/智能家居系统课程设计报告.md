## 3.1、自然语言处理全流程

本项目实现了一个面向智能家居场景的中文自然语言处理流程。用户输入一句中文指令后，系统会判断该指令是否属于智能家居控制范围，并从中提取房间、设备和动作，最后更新家居状态。

系统整体流程如下：

```text

        |
        v
写入 runtime/home_state.json
        |
        v
GUI 状态看板 / HTML 状态看板 / 数据分析侧车
```

## 3.2、关键技术概述

### 3.2.1 TF-IDF 字符级特征提取

中文短指令通常没有明显的空格分词，例如“打开客厅空调”由多个汉字连续组成。如果直接使用英文式分词方法并不合适。本项目使用 `TfidfVectorizer` 的字符级 n-gram 特征，将文本拆成单字、双字和三字组合。

例如：

```text
打开客厅空调
```

可以提取出如下字符特征：

```text
打、开、客、厅、空、调、打开、客厅、空调、客厅空、厅空调
```

这种方法不需要额外中文分词工具，适合课程项目中的短文本分类任务。

### 3.2.2 Logistic Regression 分类模型

项目使用 Logistic Regression 作为分类器。该模型训练速度快、实现简单、可解释性较好，适合中小规模文本分类任务。本项目共训练四个模型：

表 6 模型设计表

| 模型名称 | 作用 | 输出 |
| --- | --- | --- |
| 二分类意图模型 | 判断输入是否为智能家居控制 | 0 或 1 |
| 地点识别模型 | 识别房间位置 | 客厅、卧室、厨房等 |
| 设备识别模型 | 识别设备名称 | 灯、空调、窗帘等 |
| 动作识别模型 | 识别控制动作 | 打开、关闭 |

训练管道如下：

```python
Pipeline(
    steps=[
        ("tfidf", TfidfVectorizer(analyzer="char", ngram_range=(1, 3))),
        ("clf", LogisticRegression(max_iter=1000, class_weight="balanced")),
    ]
)
```

### 3.2.3 意图识别与槽位填充

意图识别用于判断用户想做什么，槽位填充用于提取完成该意图所需要的参数。在本项目中，常见语义结构如下：

```json
{
  "intent": "device_control",
  "location": "客厅",
  "device": "空调",
  "action": "打开"
}
```

其中：

- `intent` 表示指令类型；
- `location` 表示设备所在房间；
- `device` 表示被控制设备；
- `action` 表示打开或关闭。

如果一句话包含多个设备动作，系统会生成多个控制命令。例如“把客厅和卧室的灯都关了”会生成两个命令：关闭客厅灯、关闭卧室灯。

### 3.2.4 Out-of-Scope 拒识

真实用户输入不一定都属于智能家居控制。例如：

```text
打开原神
关闭微信
播放音乐
帮我查天气
```

如果系统把这些语句错误识别为家居控制，就可能造成误操作。因此本项目加入了 Out-of-Scope 拒识机制。系统会先检查文本中是否同时包含可控设备词和动作词，再结合二分类模型置信度进行判断。如果不满足条件，则返回非家居控制。

### 3.2.5 场景模式与规则推理

除了普通设备控制，项目还加入了场景模式和环境语义规则。例如：

表 7 场景模式表

| 场景 | 用户说法 | 控制动作 |
| --- | --- | --- |
| 离家模式 | 我出门了、离家模式 | 关闭客厅灯、电视、空调等 |
| 回家模式 | 我回来了、到家了 | 打开客厅灯、客厅空调等 |
| 睡眠模式 | 睡觉了、晚安 | 关闭客厅设备，调整卧室设备 |
| 观影模式 | 看电影、电影模式 | 打开电视，关闭窗帘和灯 |

环境语义规则示例：

表 8 环境语义规则表

| 用户输入 | 系统推理 |
| --- | --- |
| 客厅有点热 | 打开客厅空调 |
| 书房太暗了 | 打开书房灯 |
| 厨房太闷了 | 打开厨房油烟机 |
| 卫生间有点闷 | 打开卫生间排风扇 |

### 3.2.6 GUI、状态文件与 HTML 看板

系统使用 Tkinter 实现桌面图形界面。用户可以在输入框中输入指令，点击“分析指令”后，系统会显示指令类型、置信度、位置、设备和动作。解析结果会写入 `runtime/home_state.json`，状态看板再读取该文件展示当前设备状态。

HTML 看板由 `src/status_dashboard.py` 生成，页面每 2 秒自动刷新，便于在浏览器中查看当前家居状态。

### 3.2.7 可选 API 大模型解析

项目默认使用传统机器学习模型和规则解析，保证离线也能运行。同时项目提供 `src/api_llm.py`，可选接入 DeepSeek/OpenAI-compatible API，让 API 模型输出 JSON 语义帧。系统会校验 API 返回的房间、设备和动作，只接受项目中定义过的控制对象。

## 4.5 指令解析模块实现

指令解析模块位于 `src/intent_engine.py`。该模块定义了两个核心数据结构：

```python
@dataclass(frozen=True)
class ControlCommand:
    location: str
    device: str
    action: str


@dataclass
class ParseResult:
    text: str
    is_control: bool
    confidence: float
    location: str | None = None
    device: str | None = None
    action: str | None = None
    intent: str = "out_of_scope"
    scene: str | None = None
    commands: list[ControlCommand] = field(default_factory=list)
```

`ControlCommand` 表示一条可执行设备控制命令，`ParseResult` 表示完整解析结果。

解析流程如下：

1. 去除输入文本首尾空格；
2. 对床帘、排气扇、关上、开一下等口语词进行归一化；
3. 判断是否命中场景模式；
4. 判断是否命中环境语义规则；
5. 判断是否属于家居控制范围；
6. 使用二分类模型计算置信度；
7. 提取房间、设备和动作；
8. 根据“所有、全部、全屋”等范围词生成批量命令；
9. 返回 `ParseResult`。

普通指令示例：

```text
输入：打开客厅空调
输出：location=客厅，device=空调，action=打开
```

批量指令示例：

```text
输入：关闭所有空调
输出：关闭客厅空调、关闭卧室空调、关闭书房空调、关闭餐厅空调
```

场景指令示例：

```text
输入：我出门了
输出：离家模式，生成多个设备关闭动作
```

## 4.7 家居状态同步与看板实现

家居状态同步模块位于 `src/state_bridge.py`。系统使用 `runtime/home_state.json` 保存设备状态。

状态文件示例：

```json
{
  "updated_at": "2026-06-13T20:00:00",
  "last_command": "打开客厅空调",
  "last_result": "客厅空调开启",
  "rooms": {
    "客厅": {
      "灯": false,
      "空调": true,
      "电视": false,
      "窗帘": false,
      "风扇": false
    }
  }
}
```

当解析结果为有效控制指令时，系统会遍历 `commands` 中的每一条命令，并修改对应房间和设备的开关状态。这样主界面、独立状态看板、HTML 看板和数据分析模块都可以通过同一个状态文件进行联动。

HTML 看板由 `src/status_dashboard.py` 生成，文件位置为：

```text
runtime/home_status.html
```

页面会展示当前开启设备数量、最近一次操作、各房间设备状态，并每 2 秒自动刷新。

作为组长，我负责了项目的总体架构设计、核心技术开发和最终集成联调。在项目启动阶段，我先梳理了系统的数据流：用户输入→文本归一化→意图识别→槽位填充→状态更新→看板展示，然后根据数据流将系统拆分为数据集、模型训练、意图解析、状态同步、GUI界面和扩展模块六个部分。这个拆分让后续的并行开发成为可能，每个成员可以独立完成自己的模块。

在联调阶段，我遇到了模块间数据格式不一致的问题，例如意图引擎输出的 `ControlCommand` 和状态同步模块期望的数据结构不完全匹配。通过统一 `ParseResult` 和 `ControlCommand` 两个数据类，明确了各模块间的接口约定。这次实训让我深刻体会到，实际项目中的自然语言处理不能只依赖单一算法，需要结合具体业务场景进行工程化设计，同时模块化架构和清晰的接口定义对团队协作至关重要。
