# 新手读代码指南：名词解释与主流程

这份文档是给刚开始学 Python、机器学习和智能家居项目的新手看的。不要先从所有代码细节开始啃，建议先理解“程序分成哪几块，每块干什么”。

## 先按这个顺序读

1. 先读 `run_app.py`  
   它是桌面版启动入口，只负责调用 `src/app.py`。

2. 再读 `src/app.py`  
   它负责窗口、按钮、输入框、结果展示。你要重点看这些函数：
   - `analyze_text()`：点击“分析指令”后执行。
   - `show_result()`：把解析结果显示到界面，并更新家居状态。
   - `refresh_home_panel()`：刷新状态看板。

3. 再读 `src/intent_engine.py`  
   它是理解中文指令的核心。你要重点看：
   - `parse()`：总流程入口。
   - `_normalize_terms()`：把口语词改成标准词。
   - `_match_scene()`：识别“我出门了、睡觉了”等场景。
   - `_match_environment_intent()`：识别“太热、太暗、太闷”。
   - `_build_commands()`：生成真正要执行的动作列表。

4. 再读 `src/state_bridge.py`  
   它负责把解析结果写进 `runtime/home_state.json`。这个项目没有接真实家电，所谓“执行控制”就是修改这个 JSON 状态文件。

5. 最后读训练和大数据部分  
   - `src/data_builder.py`：生成训练数据。
   - `src/trainer.py`：训练模型。
   - `linux_bigdata/`：把状态变化变成事件，再做统计分析。

## 一句话从输入到执行的完整流程

以“打开客厅灯”为例：

1. 用户在 GUI 输入框输入：`打开客厅灯`。
2. 点击“分析指令”按钮。
3. `src/app.py` 的 `analyze_text()` 被调用。
4. `analyze_text()` 调用 `self.engine.parse(text)`。
5. `src/intent_engine.py` 的 `parse()` 开始分析：
   - 找到地点：`客厅`
   - 找到设备：`灯`
   - 找到动作：`打开`
   - 生成动作：`ControlCommand("客厅", "灯", "打开")`
6. `parse()` 返回 `ParseResult`。
7. `src/app.py` 的 `show_result()` 把结果显示到界面。
8. `show_result()` 调用 `src/state_bridge.py` 的 `apply_parse_result()`。
9. `apply_parse_result()` 把 `runtime/home_state.json` 里的 `客厅 -> 灯` 改成 `true`。
10. GUI 看板和 HTML 看板重新读取状态文件，显示客厅灯已开启。

## 本项目最关键的 3 个数据结构

### 1. ControlCommand

`ControlCommand` 表示一条真正要执行的设备动作。

```python
ControlCommand("客厅", "灯", "打开")
```

意思是：把客厅的灯打开。

### 2. ParseResult

`ParseResult` 表示一次完整解析结果。它既给 GUI 展示，也给状态同步模块执行。

它里面有：

| 字段 | 意思 |
| --- | --- |
| `text` | 用户原始输入 |
| `is_control` | 是否是家居控制 |
| `confidence` | 置信度 |
| `location` | 地点 |
| `device` | 设备 |
| `action` | 动作 |
| `intent` | 意图类型 |
| `reason` | 拒识原因 |
| `message` | 给用户看的结果摘要 |
| `commands` | 真正要执行的动作列表 |

### 3. home_state.json

`runtime/home_state.json` 保存当前所有设备开关状态。

简化后大概是：

```json
{
  "last_command": "打开客厅灯",
  "last_result": "客厅灯开启",
  "rooms": {
    "客厅": {
      "灯": true,
      "空调": false
    }
  }
}
```

`true` 表示开启，`false` 表示关闭。

## 为什么代码里既有规则又有模型

因为不同类型的句子适合不同方法。

规则适合处理确定表达：

| 输入 | 规则如何处理 |
| --- | --- |
| `我出门了` | 直接触发离家模式 |
| `厨房太闷了` | 推理为打开厨房油烟机 |
| `关闭所有空调` | 展开为多个房间的空调关闭 |

模型适合处理普通分类：

| 任务 | 模型负责 |
| --- | --- |
| 二分类 | 判断是不是家居控制 |
| 地点分类 | 判断客厅/卧室/厨房等 |
| 设备分类 | 判断灯/空调/电视等 |
| 动作分类 | 判断打开/关闭 |

简单说：规则保证关键场景稳定，模型补充普通句子的分类能力。

## 为什么要拒识非家居指令

如果不拒识，用户说“打开原神”“打开微信”时，程序可能误以为“打开”就是设备控制。

所以 `intent_engine.py` 做了两层保护：

1. 规则粗筛：句子里必须包含项目支持的家居设备和动作。
2. 模型置信度：二分类模型不够确定时，不执行任何设备动作。

## 为什么 API 大模型结果还要校验

大模型可能会输出不存在的设备组合，例如：

```json
{"location": "厨房", "device": "空调", "action": "打开"}
```

但项目配置里厨房没有空调。  
所以 `src/api_llm.py` 的 `_validated_commands()` 会把这种动作丢掉，只保留真实存在的房间和设备组合。

## 你答辩时可以这样讲项目逻辑

本项目先通过 GUI 或控制台接收中文指令，然后由本地意图解析引擎进行语义理解。解析引擎先做口语归一化，再优先识别场景模式、环境语义和批量控制；普通指令则通过二分类模型判断是否属于家居控制，并通过槽位模型或规则提取地点、设备和动作。解析结果会被封装成 `ParseResult`，其中的 `commands` 列表交给状态同步模块执行。状态同步模块把设备开关写入 `runtime/home_state.json`，GUI 看板、HTML 看板和 Linux 大数据侧车都读取这份状态文件。大数据侧车把状态变化记录成 JSONL 事件，并进一步生成批处理统计和流式窗口统计结果。
